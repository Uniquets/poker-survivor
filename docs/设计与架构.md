# 设计与架构

玩法细则以 `RULES_V0_2.md` 为准；本文档为产品愿景、目录与架构总览。

---

## 一、愿景与范围

类幸存者高压战斗 + **有序扑克牌组**驱动自动攻击与构筑；组牌、花色、遗物改变强度与节奏。

### MVP

| 项 | 说明 |
|----|------|
| 战斗 | 移动、受伤、敌波、弹道/接触伤害、难度随时间 |
| 牌组 | 开局**选** 3 张、左到右出牌、对子/三条/四条/顺子/连对/飞机、装配间隔 |
| 成长 | 经验、升级、三选一入牌 + 若干全局强化 |
| 闭环 | 单局可玩 + 基础 UI |

当前（与仓库代码对齐）：`RunScene` 为 `project.godot` 主场景（无单独 Main）；战斗、敌人、选牌 UI、`CardRuntime` 自动出牌、`GroupDetector`、`EffectResolver` 与弹道已接通；经验/升级三选一、局外金币、20 分钟胜利等尚未实现。非 MVP：多关卡大地图、深度局外养成可先占位。

### 成功标准（方向）

新玩家约 **30 秒**内理解移动与「牌在自动打」；单局 **8～15 分钟**节奏合理；组牌与花色有可感知差异。

## 二、架构

### 1. 目标

有序牌组从左到右自动出牌 → 组牌判定 → 效果结算 → 装配间隔 → 下一轮；规则与表现分离。

### 2. 场景拓扑

```
Main（入口 / 切场景）← 当前未建；主场景直接为 RunScene
└── RunScene（单局）← 现为游戏入口
    ├── Player
    ├── EnemyManager
    ├── CombatWorld（可选）
    ├── CardRuntime（牌序、节奏、指针）
    └── UI
MetaScene（局外，MVP 可先占位）
```

### 3. 脚本模块

| 路径 | 职责 |
|------|------|
| `scripts/cards/` | 牌资源、运行时、组牌检测、花色与全局被动；**全局强化**（`GlobalAugmentState` / `GlobalAugmentRule` / `PlayAugmentSnapshot`，解析前写入 `PlayContext`） |
| `scripts/combat/` | 伤害、命中、敌我属性、投射物、跑图主控；**`CombatPlayer` / `EnemyManager`** 在进树时注册进程内单例（`get_*` + `static var`，单局各一实例），供索敌与效果执行读取；弹道首发索敌由 **`CombatEffectRunner`** 调 **`TargetConfirmDefault.resolve_global`**（`PlayEffectCommand.lock_target_kind` 可选覆盖）；**`Projectile`** 经 **`configure_from_volley`** 在 `add_child` 前注入目标/直线参数与伤害 |
| `scripts/core/` | 通用类型、常量、调参；**`TargetConfirmDefault`** 索敌方案与 `resolve` / `resolve_global`，返回内嵌类 **`TargetConfirmResult`**（强类型字段，替代字典）；**`GameToolSingleton`**（Autoload）世界飘字等；**`ProjectileToolSingleton`**（Autoload）多发弹道发射几何（并行/角度扇），无玩法判定 |
| `scripts/ui/` | HUD、选牌、手牌展示等 |
| `scripts/meta/` | 存档、局外成长（待扩展） |

### 4. 组牌数据流

1. `CardRuntime` 取下一播放单元（单张组或组合）。  
2. `GroupDetector` 给出本步牌列表与牌型。  
3. **当前实现**：`AutoAttackSystem` 使用 `PlayEffectResolver` → `PlayPlan` → `CombatEffectRunner`；非关键张（非 §10）仍经内置 `EffectResolver` 再转为弹道命令。  
4. **演进方向（§7）**：继续将更多叠层拆为 **Contributor**，花色与遗物与关键张的衔接见 `RULES_V0_2.md` §10 说明。  
5. 表现层（弹道、范围、召唤物等）由命令驱动，规则层不直接操作场景节点。

### 5. 组牌不变量（摘要）

从左到右、最长组优先、打出后跳过整组、一轮内不重复消费；顺子最短 3、A 不当 1、允许 Q-K-A；合法组含连对、飞机（见 `RULES_V0_2.md` §3）。装配间隔在**本轮最后一组牌打出后**开始（不等弹道命中，见 `RULES_V0_2.md` §2）。

### 6. 资源与工具

- 资源路径：`resources/cards/` 等。  
- `addons/godot_mcp` 仅供工具链；游戏逻辑不依赖其 API。

### 7. 效果解析与执行（管线 + 命令列表，规划）

本节描述**软件架构**目标形态；**玩法叠放顺序**仍以 `RULES_V0_2.md` §5 为准，实现时管线阶段须与该顺序一致。可与现有 `EffectResolver` **渐进衔接**（例如先产出 `PlayPlan` 且内嵌与 `EffectResult` 等价的摘要，再逐步把战斗侧迁到命令执行）。

#### 7.1 设计目标

- **规则在 `cards`，执行在 `combat`**：组牌、花色、遗物等只影响「解析结果」；弹道、范围伤害、召唤物只由「已解析命令」触发。  
- **易扩展**：新牌型/遗物/表现形态主要体现为**新增管线贡献者**与**新命令类型或新表现处理器**，避免在战斗入口堆叠巨型 `match(牌型)`。  
- **与 §2 装配语义一致**：区分**逻辑步完成**与**表现播完**；逻辑相命令应先于表现相执行，且**不要求**等待弹道命中再进入装配间隔（与 `RULES_V0_2.md` §2、`CardRuntime` 一致）。

#### 7.2 核心对象（命名可映射到代码）

| 对象 | 职责 |
|------|------|
| **PlayContext** | 单次出牌解析的**只读输入**：本组牌、`group_type`、手牌花色计数快照、`augment_snapshot`（全局强化汇总）等。 |
| **PlayPlan** | 解析输出：有序 **`commands`**（命令列表），以及可选 **`summary`**（供 HUD、日志、与现有 `EffectResult` 对齐的聚合字段）。 |
| **EffectCommand**（或等价基类） | 单条可执行意图：`kind`（类型标识）、**`phase`**（`logical` / `presentational`）、以及该类型所需参数（伤害、Buff、半径、`presentation_id` 等）。不含 `Node` 引用。 |

#### 7.3 解析管线（PlayEffectResolver）

- 由多个 **`EffectContributor`**（贡献者）按**固定顺序**依次对同一 `PlayPlan` 进行**追加或合并**（例如追加一条伤害命令、或提高上一条伤害的倍率、附加标签）。  
- **阶段顺序须对齐 `RULES_V0_2.md` §5**：① 基础牌效果 → ② 组牌效果 → ③ 本次攻击花色 → ④ 同花大幅强化 → ⑤ 全局被动（花色阈值、遗物等）。新增叠层时仍插入该顺序内明确位置，并在规则文档中登记。  
- 管线结束时，扑克语义应已**完全凝结**为命令与摘要；**战斗代码不应再读取「是否为顺子」等**来做玩法分支（可读 `summary` 做飘字等）。

#### 7.4 战斗执行（CombatEffectRunner）

- **第一遍**：遍历 `commands` 中 `phase == logical` —— Buff、资源、计数器、与「本步逻辑完成」相关的状态。  
- **第二遍**：遍历 `phase == presentational` —— 生成弹道、粒子、短时 `Area2D`、召唤物场景等；**同步发起、异步演化**，不阻塞下一组牌的逻辑边界（与 §2 一致）。  
- **表现路由**：按 `kind` 或 `presentation_id` 将命令交给 **`PresentationHandler`**（或等价注册表）。新增散弹、圆形 AoE、召唤等 = **新处理器 +（通常）新场景/对象池**，不必修改 Runner 的双循环结构。

#### 7.5 扩展新效果（检查清单）

1. 在 `RULES_V0_2.md` / 测试用例中确认玩法定义与叠放顺序（若涉及新组牌类型须先更新规则文档）。  
2. 在 **`scripts/cards/`** 增加或调整 **Contributor**，向 `PlayPlan` 写入命令或修改已有命令字段。  
3. 定义或复用 **`EffectCommand`** 子类（或窄 `Resource`），填好 `phase` 与参数。  
4. 在 **`scripts/combat/`** 为 `logical` / `presentational` 增加 **dispatch** 分支或注册 **Handler**。  
5. 可选：为新表现增加 `PackedScene`，由 Handler 实例化；伤害/状态通过战斗侧统一 API 回写。

#### 7.6 与现有代码的衔接

- 当前 **`EffectResolver` + `EffectResult`** 可视为 `PlayPlan.summary` 的初版或并行输出；迁移时可先双轨（同一解析过程既写 `EffectResult` 又写 `commands`），再收敛战斗消费端。  
- **`scripts/ui/`** 仍只读状态或 `summary`，不写组牌与效果判定。

#### 7.7 全局强化（框架）

- **`GlobalAugmentState`**（由 `AutoAttackSystem` 持有）：**类型 1** 永久拾取（如 BOSS 三选一后 `grant_permanent_volley_plus_one`）与 **类型 2** 可注册规则（花色手牌档位、本组花色张数等），后续可增字段/新 `ApplicationKind`。  
- 每次解析前 `build_snapshot(ctx)` → **`PlayContext.augment_snapshot`**（`PlayAugmentSnapshot`）；**`PlayEffectResolver`** 在拼好 `PlayPlan` 后对 `PROJECTILE_VOLLEY` / `EIGHT_EXPLOSIVE_VOLLEY` 的 `count` 及 `LASER_DUAL_BURST` 的激光道数做线性加成并重算 `estimated_enemy_damage`。  
- 与 `RULES_V0_2.md` §7「牌组花色累计档位」可并存；本框架示例规则基于 **`PlayContext.global_suit_counts`（当前手牌各花色张数）**，叠乘顺序以规则文档为准，实现侧在 `GlobalAugmentRule` 内保持单一职责。  
- 入口：`RunScene.grant_global_permanent_volley_bonus()` 或 `AutoAttackSystem.grant_permanent_volley_plus_one()`；内置花色示例可通过 `GlobalAugmentState.REGISTER_BUILTIN_SUIT_VOLLEY_EXAMPLES` 关闭。
