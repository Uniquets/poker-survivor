# 工程规范

模块职责边界与玩法不变量见 `.cursor/rules/cursor-project.mdc` 与 `设计与架构.md`。

---

## 一、代码与目录

- 脚本 `snake_case.gd`，类名 `PascalCase`，成员 `snake_case`。  
- 玩法与组牌逻辑在 `scripts/cards/`；战斗消费已解析效果在 `scripts/combat/`；UI 只读状态与发意图，见 `设计与架构.md` 第三节。  
- **效果展开（目标形态）**：`scripts/cards/` 产出 **`PlayPlan` / 命令列表**（或等价物）；`scripts/combat/` **只执行命令**，不在此写组牌/牌型规则；管线顺序与两相执行见 `设计与架构.md` **§7**。  
- **中文注释（强制）**  
  - **每个函数、方法**（含 `func`、`static func`、`_ready` / `_process` 等内置回调、信号连接到的槽）：须在函数前或紧邻声明处用**中文**说明职责、关键参数/返回值、副作用或与文档的对应关系。  
  - **全局变量与脚本/类作用域字段**：Autoload 成员、以及**非函数体内**的 `var` / `const`（含类内成员字段）：均须有**中文**注释说明含义、单位或约束。  
  - **函数体内局部变量**不强制逐条注释；非显而易见的分支、状态机、魔法数、调用顺序等仍须用中文写清意图与约束。  
  - **逻辑分支处加强注释（强制）**：`if` / `elif` / `else`、`match` 各分支、多路 `return` 等形成互斥或顺序依赖时，须在分支前或分支首行用中文说明条件语义、走该路径的原因及与其它分支的边界；避免无说明的裸分支（单行早退可一句简注）。  
  - 注释须简短有信息；勿复述代码字面，勿堆砌无信息注释。  
- Godot 4 语法与标准库；常驻 UI 以场景预制体为主，避免大量代码动态拼 UI。  
- 卡牌、敌人等用 `Resource`，建议含 `schema_version`；路径如 `resources/cards/`、`resources/enemies/`。

---

## 二、工程边界

- 未经明确要求，不修改 `addons/` 及 `project.godot` 自动加载、插件与 MCP 配置。  
- 重命名对外脚本类须说明迁移方式。  
- 默认单次任务少量文件；超出先说明原因；不做任务外顺手重构。  
- 所改脚本无解析/运行时错误；合并前至少在编辑器打开并跑通主流程一次。  
- 卡牌/组牌改动须对照 `RULES_V0_2.md` 与 `测试与进度.md` 中的用例章节。

---

## 三、Agent 实现任务

- 只完成约定任务；非要求不重构。  
- 每次实现类回复须包含：**已改文件列表**、**实现要点（≤5 条）**、**手动验证步骤**、**风险或后续项**。  
- 文档服从顺序见 `.cursor/rules/cursor-project.mdc`；与文档冲突时先改文档再改代码。  
- 组牌改动最小自检：Q-K-A 合法顺子；A-2-3 不成立；一轮内每张牌只属于一组，打出后指针跳过整组。
