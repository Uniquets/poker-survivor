# 工程规范

模块职责边界与玩法不变量见 `.cursor/rules/cursor-project.mdc` 与 `详细设计.md`（**完全体内容**见该文档第二部）。

---

## 一、代码与目录

- 脚本 `snake_case.gd`，类名 `PascalCase`，成员 `snake_case`。  
- 玩法与组牌逻辑在 `scripts/cards/`；战斗消费已解析效果在 `scripts/combat/`；UI 只读状态与发意图，见 `详细设计.md` 第 9、13 节及第二部内容规格。  
- **效果展开（目标形态）**：`scripts/cards/` 产出 **`PlayPlan` / 命令列表**（或等价物）；`scripts/combat/` **只执行命令**，不在此写组牌/牌型规则；管线顺序与两相执行见 `详细设计.md` **第 9 节**。  
- **中文注释（强制）**  
  - **每个函数、方法**（含 `func`、`static func`、`_ready` / `_process` 等内置回调、信号连接到的槽）：须在函数前或紧邻声明处用**中文**说明职责、关键参数/返回值、副作用或与文档的对应关系。  
  - **全局变量与脚本/类作用域字段**：Autoload 成员、以及**非函数体内**的 `var` / `const`（含类内成员字段）：均须有**中文**注释说明含义、单位或约束。  
  - **函数体内局部变量**不强制逐条注释；非显而易见的分支、状态机、魔法数、调用顺序等仍须用中文写清意图与约束。  
  - **逻辑分支处加强注释（强制）**：`if` / `elif` / `else`、`match` 各分支、多路 `return` 等形成互斥或顺序依赖时，须在分支前或分支首行用中文说明条件语义、走该路径的原因及与其它分支的边界；避免无说明的裸分支（单行早退可一句简注）。  
  - 注释须简短有信息；勿复述代码字面，勿堆砌无信息注释。  
- Godot 4 语法与标准库；常驻 UI 以场景预制体为主，避免大量代码动态拼 UI。  
- 卡牌、敌人等用 `Resource`，建议含 `schema_version`；路径如 `resources/cards/`、`resources/enemies/`。
- 关键配置资源须有可测试的显式校验入口；当牌型目录、效果规格、默认预制体或必需音效缺失时，回归测试应通过清晰错误信息失败，禁止只依赖运行时 warning 或静默跳过命令。
- 牌型目录内的并行弹道与航点弹道必须能解析到完整表现资源：预制体、发射音、首段命中音、穿透命中音、换向命中音均须来自条目自身、`DEFAULT` 同类规格或目录默认字段；陨石场景仍由陨石规格显式配置。

## 二、工程边界

- 未经明确要求，不修改 `addons/` 及 `project.godot` 自动加载、插件与 MCP 配置。  
- 重命名对外脚本类须说明迁移方式。  
- 默认单次任务少量文件；超出先说明原因；不做任务外顺手重构。  
- 所改脚本无解析/运行时错误；合并前至少在编辑器打开并跑通主流程一次。  
- **Godot MCP Pro 测试（逻辑代码必备）**：每次修改 **`scripts/cards/`、`scripts/combat/`、与运行强相关的核心逻辑**（不含纯资源/纯文案）后，在开发环境已配置并可用 **`user-godot-mcp-pro`（Godot MCP Pro）** 的前提下，须通过 MCP 做与**本次改动相称**的验证（如 `run_test_scenario` / `execute_game_script` / 模拟输入等，见 `.cursor/rules/cursor-project.mdc`）；MCP 不可用时的豁免写法同该规则。  
- **快测必做其一（H 菜单 ↔ MCP）**：涉及**同点数弹道/战斗表现**（尤其点数 **2**）时，须在交付前完成**至少一种**快速验证：**①** 在 `RunScene`（`enable_test_functions`）内用 **H 测试菜单** 自选花色与点数 **2**，分别凑齐单张 / 对子 / 三条 / 四条并打出或核对 `PlayEffectResolver` 输出是否符合预期；**②** 或在 **Godot MCP Pro** 中用 `execute_game_script` 等价构造牌组并调用解析/执行路径（见 `docs/测试与进度.md`「H 菜单与 MCP 快测」）。并须**检查编辑器诊断/报错**（MCP 的 `get_editor_errors` 或编辑器「输出/错误」面板）。  
- 卡牌/组牌改动须对照 `详细设计.md` 与 `测试与进度.md` 中的用例章节。

## 三、Agent 实现任务

- 只完成约定任务；非要求不重构。  
- **Godot MCP Pro + H 菜单快测（必须遵守，与 `.cursor/rules/cursor-project.mdc` 硬性条一致）**：在已配置且会话可调 **`user-godot-mcp-pro`** 的前提下，为完成任务**应优先**用 **Godot MCP Pro** 做打开工程、场景/脚本检查、运行或冒烟；**每次修改逻辑代码后，必须通过 Godot MCP Pro 做与本次改动路径相称的验证**（含运行态断言/模拟操作，禁止「只启动不操作」作为唯一验证）；**在修改 Godot 相关代码或场景后，还须通过 MCP 做编译/诊断类验证**（MCP 不可用时须在交付说明的「风险或后续项」中写明，并给出编辑器内手动验证步骤）。**同点数 2 等战斗表现改动**：须按 `测试与进度.md` 完成 **H 菜单快测或 MCP 等价快测**至少其一，并在回复中写明**自动验证结果**与**编辑器无新增报错**（或已处理的报错说明）。
- 每次实现类回复须包含：**已改文件列表**、**实现要点（≤5 条）**、**自动验证结果**（含 Godot MCP Pro / H 菜单快测结论与编辑器报错检查）、**手动验证步骤**、**风险或后续项**。  
- 文档服从顺序见 `.cursor/rules/cursor-project.mdc`（以 `详细设计.md` 为首要权威）；与文档冲突时先改文档再改代码。  
- 组牌改动最小自检：Q-K-A 合法顺子；A-2-3 不成立；一轮内每张牌只属于一组，打出后指针跳过整组。
